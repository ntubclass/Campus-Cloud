PVE Placement Algorithm

VM / LXC 分配流程與 Storage 排序邏輯圖解（含 CPU / Disk Overcommit 與強制平均分配）

① 輸入

VM Template CPU / RAM / Disk
resource_type: qemu | lxc
storage_scope: any | local | shared
preferred_storage: (可選)
PVE Nodes CPU / RAM 實體容量
node priority (1–10)
strategy: dominant | priority
Overcommit 設定
cpu_overcommit_ratio (預設 2.0，範圍 1.0–8.0)
disk_overcommit_ratio (預設 1.0，範圍 1.0–5.0)
兩者皆先塞正常空間，不足才超額
↓建立執行狀態

② Shared Pool 容量只計一次

_build_servers() — Shared Pool Registry
Local Pool（各節點獨立）
pve-a → local-lvm
pve-b → local-lvm
pve-c → local-lvm
✓ 各自獨立 avail_gb、placed_count
Shared Pool（NFS / Ceph…）
pve-a ──┐
pve-b ──┼──► ceph-pool (同一物件)
pve-c ──┘
✓ 扣容量所有節點同步

每個 pool 初始化時計算 overcommit_avail_gb = total_gb × (ratio − 1.0)，placed_count = 0，overcommit_placed_count = 0
_clone_servers() 用 pool.storage（名稱）識別相同 shared pool，rebalance 搜尋也保持共用物件

↓每小時模擬

③ 節點選擇 — 兩階段 CPU Overcommit

Phase 1 — 正常（Physical CPU）
CPU ✅ 不超額
total_cpu − used_cpu ≥ request
只用實體核心數
RAM
usable = 實體 × 0.9
used + request ≤ usable
Disk
有 pools → 看 pool avail_gb
(含 overcommit_avail_gb)

Phase 1 找到候選節點 → 直接進入排序，不進入 Phase 2

Phase 2 — CPU Overcommit（Phase 1 全部失敗才觸發）
CPU 🔴 超額
total_cpu × ratio − used_cpu ≥ request
ratio 可設 1.0–8.0
RAM
同 Phase 1
Disk
同 Phase 1

超額放置的 VM 在計算表格中顯示 CPU OC 標籤

任一條件不滿足 → 此節點排除，不進入排序

↓通過篩選的節點進入排序

④ 節點排序（Sort Key，越小越優先）UPDATED

Strategy: dominant_share_min
# placement_count 第一優先 → 強制輪流分配 (placement_count, # 放置數最少的節點優先 ★ score, # dominant_share + penalty avg_share, # 平均 share 微調 cpu_share, # CPU 占比 name) # 打平用

不再使用 score_bucket 分組——直接以放置次數為首要排序鍵，
確保異質叢集（大小節點混合）也能平均分配。
_can_fit() 作為安全網，防止小節點被超載。

Strategy: priority_dominant_share
# 同優先權節點先平均，再比負載 (node.priority, # 1=最優先 placement_count, # 同 priority 先平均分散 score, # 同數量再比負載分數 avg_share, cpu_share, name)
score = projected_dominant_share + resource_penalty + migration_cost
cpu_share × 1.0 + cpu_contention × 2.0 ← 超過 0.7 開始 penalty memory_share × 1.2 + memory_overflow × 5.0 ← 超過 policy 容量重罰 disk_share × 1.5 + disk_contention × 1.5 ← 超過 0.75 開始 penalty gpu_share × 3.0 ← GPU 稀缺，權重最高 loadavg_pen × 0.9 ← 實際負載 soft penalty（不直接拒絕）

cpu_share 基準 = used_cpu / (total_cpu × cpu_overcommit_ratio)，使用者設定的 ratio 影響 dominant share 計算

↓選出最佳節點後

⑤ Storage Pool 選擇 — 兩階段（Normal → Disk Overcommit）

Phase 1 — 正常放置（avail_gb ≥ disk_gb）
0
Hard Filter：active + enabled + can_vm/can_lxc + avail_gb ≥ disk_gb
ISO / Backup pool 自動排除；enabled=false 排除
1
Preferred Storage Name（exact match 直接勝出，跳過排序）
2
Scope 偏好（軟規則）
local → score 0 shared → score 0 不符偏好 score=1，仍可 fallback
3
Speed Tier
NVMe = 0 SSD = 1 HDD = 2 數字越小越優先
4
User Priority（1=最優先，1–10）
5
placed_count（同速度同優先權的 pool 輪流接收）
計數少的 pool 優先，確保計數型平均分配
6
剩餘空間最多（−avail_gb，打平最後手段）
# Phase 1 sort key（avail_gb ≥ disk_gb 的 pool） min(normal, key=lambda p: ( scope_score(p), # 0 or 1 SPEED_RANK[p.speed_tier], # 0/1/2/3 p.user_priority, # 1–10 p.placed_count, # 平均分散 -p.avail_gb, # 剩餘最多 ))
Phase 2 — Disk Overcommit（Phase 1 無候選才觸發）
候選條件：avail_gb + overcommit_avail_gb ≥ disk_gb
（overcommit_avail_gb = total_gb × (disk_overcommit_ratio − 1.0)）
# Phase 2 sort key（超額候選） min(overcommit, key=lambda p: ( p.overcommit_placed_count, # 超額次數少的優先（平均分散） scope_score(p), SPEED_RANK[p.speed_tier], p.user_priority, -(p.avail_gb + p.overcommit_avail_gb), ))

放置後在計算表格顯示 OC 標籤；preferred_storage_name 在超額候選中也有效

↓選定 pool 後

⑥ 容量扣除

Local Pool — 正常放置 pool.avail_gb −= disk_gb
pool.placed_count += 1
server.used_disk += disk_gb
只影響該節點的 disk budget
Shared Pool — 正常放置 pool.avail_gb −= disk_gb
pool.placed_count += 1
server.used_disk 不動
avail_gb 是共用物件 → 所有節點同步
Disk Overcommit 放置 from_normal = max(pool.avail_gb, 0)
oc_used = disk_gb − from_normal
pool.avail_gb = 0（先耗盡正常空間）
pool.overcommit_avail_gb −= oc_used
pool.overcommit_placed_count += 1
還原時 avail_gb / overcommit_avail_gb 各自恢復
CPU Overcommit 放置 is_cpu_overcommit = (used_cpu + request) > total_cpu
server.used_cpu += request（無論是否超額）
CPU 沒有獨立 bucket，超額直接累加 used_cpu
計算表格顯示 CPU OC 標籤
↓若無節點能容納

⑦ Local Rebalance（最多 2 次搬移）

_local_rebalance() — 搜尋最多 2-move 的騰出空間方案
1. 找 所有已放 VM 中，哪些搬到別的節點可以騰出空間
2. 搬移目標同樣用 _placement_sort_key（含 use_node_priority 與 placement_count 平均分散）
3. 搬移前先用 _can_fit() 驗證目標節點確實有容量
4. 最多搜尋 2 層深度（避免搜尋爆炸）
5. 搬移時 pool.placed_count / overcommit_placed_count 正確還原與重套用
6. 找到方案 → 套用並回報「rebalanced N VMs」
7. 找不到方案 → 此 VM 標記 no_fit

Key Design Decisions

兩階段 CPU Overcommit Phase 1 優先使用實體核心（不超額）；
Phase 2 只在所有節點物理 CPU 皆不足時觸發，
不混入 Phase 1 候選節點排序。
placement_count 強制平均分配 UPDATED
兩種策略的節點排序皆以 placement_count 為首要鍵（priority 模式則 node.priority 在前）。
不再使用 score_bucket 分組，避免異質叢集（大小節點）集中在大節點。
_can_fit() 作為安全網防止小節點超載。
Storage：placed_count / overcommit_placed_count 確保 pool 間輪流。
兩階段 Disk Overcommit Phase 1 優先正常 avail_gb；
Phase 2 以 overcommit_placed_count 為首要排序，
確保超額負載平均分散，不集中在單一 pool。
Shared Pool 不重複計算 同一個 Python 物件被多個節點引用（用 pool.storage 名稱識別）。
一個節點扣容量，其他節點立刻看到變化。
_clone_servers() 用名稱鍵而非 id() 追蹤共用 pool。
全部 Soft Rule（Scope / Speed） Scope / Speed / Priority 都是排序鍵，
不是 hard exclusion。偏好不符仍可放置，
不產生假失敗。
三層獨立控制 node.priority → 去哪台節點
storage.user_priority → 用哪個 pool
overcommit_ratio → 超額門檻
三層分開，互不干擾。
