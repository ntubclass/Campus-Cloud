# AI 功能（模板設計與金鑰分配）路由整合與維運收斂報告（2026-04-15）

## 1. 分析目標

本報告聚焦你負責的兩塊：

1. 模板設計與推薦流程是否已收斂為單一路徑。
2. AI 金鑰申請、核發、使用是否已形成一條可維運的標準流程。

限制：僅盤點與建議，不修改程式。

---

## 2. 先講結論

目前狀態不是「很亂」，但也不是「單一單一」。

比較精準的描述是：

- 金鑰分配鏈路：已接近單一路徑（成熟度高）。
- 模板推薦鏈路：功能可用，但前後端資料載入與配置治理仍是多路並行（成熟度中）。
- 全域 AI 維運：仍是多設定來源、多上游入口，觀測面不足（成熟度中低）。

一句話總結：

「產品流程有主幹，但維運平面尚未收斂成單一標準。」

---

## 3. 路由整合現況盤點

### 3.1 金鑰分配主鏈（相對收斂）

後端主鏈：

- 申請：/api/v1/ai-api/requests
- 個人申請查詢：/api/v1/ai-api/requests/my
- 管理員審核：/api/v1/ai-api/requests/{request_id}/review
- 個人金鑰管理：/api/v1/ai-api/credentials/my
- 管理員金鑰盤點：/api/v1/ai-api/credentials
- 實際推論代理：/api/v1/ai-proxy/chat/completions

前端主鏈：

- 使用者頁：/_layout/ai-api（申請、查詢、管理）
- 管理審核頁：/_layout/ai-api-approvals
- 管理金鑰頁：/_layout/ai-api-credentials

判讀：

- 這條鏈路從申請到可使用 API key，路由主幹清楚、責任分層明確。
- 權限閘門前後端一致（admin 可進管理頁，後端 AdminUser 亦可執行審核/全域查詢）。

### 3.2 模板設計與推薦鏈（部分收斂）

後端推薦路由：

- 對話：/api/v1/ai/template-recommendation/chat
- 產生方案：/api/v1/ai/template-recommendation/recommend

前端入口：

- ApplicationRequestPage 內嵌 AiChatPanel 呼叫推薦 API。
- FastTemplatesTab 透過 virtual:templates 載入模板 JSON。

資料來源：

- 模板檔集中在 frontend/src/json。
- 後端 catalog 也讀同一批 JSON（由 system-ai.json 的 templates_dir 指向）。

判讀：

- 模板資料來源看似單一，但載入機制是前端 build/plugin 與後端 runtime 兩條路，屬於「同資料、雙管線」。

### 3.3 其他 AI 路由（旁路）

- AI Judge：/api/v1/rubric/*（群組下 teacher/admin 可用）
- PVE Advisor：/api/v1/ai/pve-advisor/recommend（後端可用，前端缺獨立可視化）

判讀：

- AI 能力不只一條主線，而是多個子域共存。
- 這是合理演進，但需要統一維運標準，不然會慢慢漂移。

## 4. 你最關心的問題：路由「可以整合」還是「很亂」？

答案：可以整合，而且已經有可整合主幹；目前問題主要在維運收斂，不在功能不存在。

### 已經整合得不錯

1. 金鑰申請與審核流程完整。
2. API key 驗證、配額、代理已串起來。
3. 後台也有審核與金鑰清單。

### 還不夠單一的地方

1. AI 設定來源至少三套並行：
   - system-ai.json + VLLM_*（模板推薦/PVE advisor）
   - core 設定中的 TEMPLATE_RECOMMENDATION_VLLM_*（rubric 仍沿用）
   - features/ai/config 的 AI_API_*（ai-proxy）
2. 推論上游存在雙入口概念：
   - 一路走 VLLM_BASE_URL
   - 一路走 AI_API_BASE_URL（對 vllm gateway）
3. 模板資料雖同源，但前後端載入時機不同，更新一致性要靠流程約束。

## 5. 風險與可能問題清單（按優先度）

### P0（高）

1. 設定分散導致變更風險
- 同一次模型切換，需要改多個設定群組。
- 容易發生某子系統仍指向舊模型或舊 base URL。

2. 維運可視化不足
- 雖有 /ai-proxy/usage/my 與 /ai-proxy/rate-limit/status 路由，
  但管理端尚未形成統一營運看板。
- 你要管理模型可用性、延遲、TPS、錯誤率、配額時，缺單一視圖。

3. 使用量資料可能不完整
- 服務內有 record_usage 函式與 AIAPIUsage 模型，
  但目前沒有看到在代理流程中被實際呼叫記錄。
- 這會直接影響 usage 統計可信度。

### P1（中）

1. 模板更新一致性風險
- 後端 catalog 使用 lru_cache，缺 reload 機制。
- 前端 virtual:templates 在 build/plugin 載入，部署後通常需重建前端才更新。
- 會出現「後端已更新、前端仍舊資料」或反向差異。

2. PVE Advisor 產品化不足
- 路由存在，但缺管理端可視化頁面與解釋層，難做營運追蹤與校正。

3. 申請有效期驗證偏寬
- API 層的 duration 是字串欄位，後端服務以特定字串判斷期限。
- 若外部呼叫傳入非預期值，可能落入預設長期有效行為。

## 6. 對你提出的三項收斂方向的落地評估

### 6.1 統一 AI 設定來源

結論：必要且優先，現在就該啟動。

建議目標：

- 單一設定模型：system-ai.json（結構化）+ .env 覆蓋（機敏值）。
- 統一命名：VLLM_BASE_URL、VLLM_API_KEY、VLLM_MODEL_NAME 與子域參數。
- 逐步淘汰重複設定鍵，避免 rubric/ai-proxy 各自維護一套。

### 6.2 明確 AI 推論平面標準

結論：必要，否則環境漂移一定發生。

建議目標：

- 指定單一正式上游（例如 vllm-API）作為唯一 production upstream。
- 其他 inference 服務標記為 lab/benchmark，從正式 compose 與正式設定分離。
- 所有路由都透過同一 upstream 抽象層，不直接各自組 URL。

### 6.3 補上運維可視化

結論：現在最短板。

建議至少補齊：

1. 模型可用性（health、model list）
2. 延遲（p50/p95）
3. token throughput（input/output tps）
4. 錯誤率（4xx/5xx、upstream timeout）
5. 配額使用（每 key、每 user、每時段）

## 7. 4 到 6 週收斂路線圖（不改架構大方向）

### 第 1 到 2 週：設定收斂

交付物：

1. AI 設定字典與責任表（哪個欄位誰擁有）
2. 單一設定模型草案（system-ai + env overlay）
3. 舊設定鍵對映表與淘汰時程

驗收：

- 同一模型切換，只需改一組標準鍵即可生效於模板推薦、ai-proxy、rubric。

### 第 3 到 4 週：推論平面標準化

交付物：

1. 正式 upstream 定版文件（例如 vllm-API）
2. 非正式路徑標註與隔離策略（lab/benchmark）
3. 路由層 upstream 統一對接檢查清單

驗收：

- 三條主 AI 路由（template recommendation、ai-proxy、rubric）在正式環境只指向一個上游入口。

### 第 5 到 6 週：運維可視化

交付物：

1. 管理頁 AI 總覽（可用性、延遲、TPS、錯誤率、配額）
2. 告警門檻（錯誤率、超時率、配額異常）
3. 日常維運手冊（排障流程、升級流程）

驗收：

- 管理員可在單頁看到所有 AI 子域健康狀態與使用壓力，不需跨頁拼接。

---

## 8. 給你的執行建議（模板設計與金鑰分配負責人視角）

1. 先抓設定收斂，不要先做更多功能頁。
2. 金鑰鏈路已經是最完整主幹，可優先作為「治理樣板」。
3. 模板鏈路下一步不是再加推薦能力，而是補「更新一致性」與「可觀測性」。
4. 對外只宣告一個正式 AI 入口，降低教學現場與維運溝通成本。
